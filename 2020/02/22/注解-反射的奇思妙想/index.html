注解+反射的奇思妙想

发表于 2020-02-22 分类于 反射 ， 注解

在 链式调用封装反射API 一文中，我们征服了 Java 反射。但是它太粗犷了，能不能细腻些？当然！本篇就带你领略下在注解的协助下，反射那些好玩的事。

注意：本篇只是注解+反射的最小化实现。只介绍这种思想，想要应用到项目中，还有很多的工作要完成。所以，仅供学习参考，切勿直接应用到项目中。

一、什么是注解

注解就像日常生活中的标签，衣服的标签上有价格；食品的标签上有价格；看的书也有价格。这类标签具有实效性，出厂时的定价就已经确定，除了促销活动，它的价格，就是签标上的数字，当我们买回来之前它有意义，对于买回来穿到身上的衣服，标签就没有意义了。

所以，我们这里说的标签，有两层含义：1. 它用来记录某些信息，比如价格；2. 它有实效性，结帐后，标签的用处就算结束。

在程序中，这个标签长什么样子？我们先来看一个简单的 @Override：

这里的 @Override 就是注解，这个注解告诉编译器，onCreate() 方法重写自父类。

二、为什么要用注解

像上面的 @Override 就为了告诉编译器，对这个方法进行安全性检查的；

在我们日常开发中，经常会遇到获取组件，并对它进行操作的。如果一个页面非常复杂，就会出现很 findViewById 的操作，这些模板代码即不优雅，写起来浪费时间。那么有没有更优雅的办法呢？

那就是本文的重点，通过反射，获取运行时注解，拿到属性上的id，然后由我们的工具来完成 findViewById 的操作。

在一个大项目中，实现一套这样的工具，节省的时间是非常乐观的，接下来进入实操部分。

下面，分别介绍下：

@Target：表示打在什么位置上，对于 Java 的类，可以大致分为：类、方法、属性 等；对应的 ElementType 分别为：TYPE、METHOD、FIELD。

@Retention：表示时效性，对于一个 Java 类，它的生命周期有三个阶段：.java 时期，.class 时期，装载到JVM使用的运行时期；对于 RetentionPolicy 分别为： SOURCE，CLASS，RUNTIME。

看了上面的解释，我们就知道 ViewById 是一个运行时的注解，并且它只能使用在属性上。

在这里，我们还可以看出，注解上可以添加注解，这个添加在注解上的注解被称为：元注解。
